# Seeder.tw 1:1 復刻標準開發工作流

本工作流旨在規範對 Seeder.tw 平台進行 1:1 能力對齊與復刻的標準 Prompt Engineering 開發閉環。在開發任何新頁面或功能模組時，AI Agent 皆須嚴格遵守以下步驟循序漸進，不可跳步。

## 階段 1：模組劃分與深度分析 (Planning)
1. **模組劃分**：將目標系統的整體平台能力，明確拆解並對應到具體的「頁面功能模組」（例如：設定頁、Webhook 訂閱、數據分析大盤等）。
2. **深度分析 (`https://seeder.tw/`)**：
   - 針對鎖定的單一模組，深入分析該網頁的**頁面交互設計 (UX/UI)**、**業務操作流程**、以及推導其**底層 API 接口實現邏輯**。
   - 提取其核心價值與必備功能點。

## 階段 2：方案對齊與架構設計 (Design)
3. **方案對比與更新**：
   - 將上述分析結果與本地現有的技術實現方案（如 `recur_design_v1.md`）進行對比。
   - 若發現現有架構無法支撐/有遺漏，**必須優先更新本地技術實現方案文檔**，包含但不限於資料表設計 (DB Schema) 或微服務 API 合約設計。確保底層能力能 1:1 承載。

## 階段 3：代碼開發與基礎自測 (Implementation)
4. **代碼開發 (Code Dev)**：嚴格依照更新後的方案，進行前後端 (Next.js + Go) 程式碼撰寫。
5. **基礎功能自測與用例補充 (UT/UK)**：
   - 開發完畢後，必須針對核心業務邏輯補充必要的測試用例 (Use Cases) 或單元測試 (Unit Tests)。
   - 確認資料流傳遞無誤，邊界條件皆有處理。

## 階段 4：本機運行與嚴格能力驗證 (Verification & Iteration Loop)
6. **運行工程 (Run Project)**：執行編譯與啟動指令，確保前後端服務正常啟動且無任何 Warning/Error 日誌。
7. **訪問頁面與差異對比 (Compare & Analyze)**：
   - 本地訪問實作頁面與 API 介面。
   - 進行**嚴格差異度對比**：與 seeder.tw 的原始功能進行比較。**視覺色彩/風格可以存在差異，但是「業務能力」、「操作路徑」與「交互回饋」必須做到 1:1 完全對齊。**
8. **自我反覆運算迴圈 (Iterate until satisfied)**：
   - 若在對比中發現能力漏接、邏輯不通或頁面無法正常使用：
     -> **更新程式碼** -> **再次進行基礎自測** -> **重新運行工程** -> **再次進行頁面對比**。
   - **此迴圈必須持續執行，直到 AI Agent 「自我審查確認滿意，且完全達到 1:1 能力對齊標準」為止。** 退回半成品或無法使用的空殼頁面是絕對不被允許的。
